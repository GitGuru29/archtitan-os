1. WHAT IS AWI?
───────────────
Autonomous Workload Intelligence is a system daemon that monitors running
processes in real-time and automatically tunes the Linux kernel, CPU governor,
memory manager, and background services — WITHOUT any user input.

Unlike manual profile switchers (gamemode, tuned, KDE Activities), AWI requires
zero configuration. The OS adapts itself like an automatic transmission.


2. WHY IT'S NEW (NOTHING LIKE THIS EXISTS)
──────────────────────────────────────────
  Windows:   Static memory compression. Treats all apps equally. Dumb.
  macOS:     App Nap + memory pressure. Only pauses background apps.
  Linux:     OOM Killer. Waits until RAM is FULL, then randomly kills apps.
  Gamemode:  User manually enables "gaming mode". Manual, single purpose.

  ArchTitan AWI:  Watches running processes, auto-detects workload type,
                  and PRE-EMPTIVELY tunes the kernel BEFORE you run out
                  of memory. Nothing like this exists on any OS.


3. HOW IT WORKS — PROCESS FINGERPRINTING
────────────────────────────────────────
The daemon maintains a lightweight lookup table that maps process names
to workload classifications:

  ┌─────────────────────────┬──────────────────────┬──────────────────────────┐
  │ Process Detected        │ Workload Type        │ Kernel Action            │
  ├─────────────────────────┼──────────────────────┼──────────────────────────┤
  │ gradle, adb             │ Android Dev          │ Max CPU, high I/O prio   │
  │ docker, containerd      │ Container Build      │ Expand cgroups, raise    │
  │                         │                      │ OOM threshold            │
  │ ollama, python torch    │ AI/ML                │ Pin GPU mem, disable swap│
  │ steam, wine             │ Gaming               │ CPU perf, GPU max clocks │
  │ code, nvim              │ Coding               │ Balanced, prioritize IDE │
  │ chromium, figma         │ Design               │ GPU for rendering        │
  │ (nothing heavy)         │ Idle                 │ CPU powersave, cache     │
  └─────────────────────────┴──────────────────────┴──────────────────────────┘

No machine learning. Just a fast, deterministic decision tree based on
process names. Lightweight, predictable, zero overhead.


4. REAL-WORLD EXAMPLE FLOW
──────────────────────────
  Developer opens VS Code + terminal
    → AWI detects: "coding workload"
    → Action: lower swappiness, prioritize IDE memory, suspend Bluetooth

  Developer launches Docker container
    → AWI detects: "heavy build workload"
    → Action: CPU to performance, increase I/O priority for Docker,
              pre-allocate swap space

  Developer closes Docker, opens Chromium + Figma
    → AWI detects: "design workload"
    → Action: release Docker memory, prioritize GPU for browser,
              restore CPU powersave


5. TECHNICAL IMPLEMENTATION
───────────────────────────
  Language:       Python (lightweight daemon)
  Process scan:   /proc filesystem polling (every 2 seconds)
  Kernel tuning:  sysctl (vm.swappiness, vm.vfs_cache_pressure)
  CPU governor:   cpufreq (performance / powersave / schedutil)
  Service mgmt:   systemctl start/stop for background daemons
  GPU control:    nvidia-smi / AMDGPU sysfs for clock management
  Cgroup mgmt:    systemd-run for per-process resource limits
  User feedback:  Waybar module showing current detected workload

  Architecture:
    titan-awi (daemon)
    ├── /etc/titan-awi/
    │   ├── fingerprints.json    (process → workload mapping)
    │   └── actions.json         (workload → kernel actions)
    ├── /usr/bin/titan-awi       (daemon binary/script)
    ├── /usr/bin/titan-awi-ctl   (CLI: titan-awi-ctl status)
    └── systemd service file
